# 产品思考

## 起点：你到底在做什么

之前的所有版本都在说"你在做一台设备"。这是对的，但不完整。

你在做的是两件事叠加：

1. **一个框架** —— 像 OpenClaw 那样：记忆系统、工具调用、多通道接入、可扩展的 Agent 架构
2. **多模态感知 + 物理实体** —— 这是 OpenClaw 永远做不到的：实时看、听、理解环境，而且有物理存在

这两件事分开看都不新鲜。框架到处都是（LangChain、OpenClaw、AutoGPT）。多模态模型也到处都是（GPT-4o、MiniCPM-o）。

但**框架 + 多模态感知 + 物理实体 + 完全本地**这个组合，目前没有人做到。

这就是你的东西。让我从这个组合出发推演。

---

## 推演一：OpenClaw 做到了什么，做不到什么

OpenClaw（339k stars）证明了一件事：**个人 AI 助手 + 持久记忆 + 工具调用 + 多平台接入**有巨大需求。

OpenClaw 的能力：
- 记忆系统（auto-capture, auto-recall, semantic search）
- 工具调用（浏览器、文件、代码执行）
- 多平台通道（WhatsApp、Telegram、Discord、CLI）
- 人格系统（类似 SOUL.md）
- 插件架构

OpenClaw 做不到的：

**1. 它没有感知。** OpenClaw 只知道你在聊天窗口里输入了什么。它不知道你在开会、不知道有人走到你面前、不知道你面前的桌上放着什么文件、不知道你的语气是焦虑还是轻松。它是一个**纯文本世界里的 AI**。

**2. 它没有物理存在。** OpenClaw 住在你的聊天 App 里。你打开 Telegram 才能跟它说话。它不在你的物理空间里。它不是"在场"的。

**3. 它的大脑在云端。** OpenClaw 调用 Claude/GPT-4 做推理。你的对话、你的记忆、你的工具调用的上下文全部发到了 OpenAI/Anthropic 的服务器。

这三个限制不是 bug，是 OpenClaw 架构的根本约束。它是一个**虚拟的、文本的、云端的**个人 AI。

你的东西是：**物理的、多模态的、本地的**个人 AI。

这不是"更好的 OpenClaw"。这是一个不同象限的东西。

## 推演二：框架 + 多模态感知的组合创造了什么

分开看：

- 纯框架（OpenClaw）：文字进、文字出。用户主动输入，AI 被动响应。
- 纯多模态模型（GPT-4o Vision）：能看图、能听音，但没有记忆、没有工具、没有持续运行。

组合起来会发生什么？

**AI 从"你来找它"变成"它跟着你"。**

OpenClaw 模式：你打开 Telegram → 输入问题 → 等回答 → 关掉。AI 存在于一个聊天窗口里。

你的模式：AI 就在你桌上。它持续看、持续听。你不需要打开任何 App。你不需要主动发起对话。它知道你周围正在发生什么。你需要的时候直接说话，它已经有上下文了。

这个差异看起来只是"交互方式不同"，但推演下去会发现它改变了几件根本的事：

**1. 信息采集从主动变成被动。**

OpenClaw 的记忆来源是你发给它的消息。你不发，它就不知道。

你的设备的记忆来源是它持续感知到的环境。你不需要做任何操作。你正常工作、正常开会、正常讨论，它在旁边自动采集有价值的信息。

这意味着你的记忆系统的覆盖面天然比 OpenClaw 大几个数量级——因为你每天说的话远远多于你主动发给 AI 的消息。

**2. 上下文从文本变成场景。**

OpenClaw 知道你发了什么文字。你的设备知道发生了什么事。

"张总说下周五交付"——OpenClaw 能记住这句文字。你的设备还知道：这是在一个会议上说的、当时张总拿着一份项目计划、你的表情显示你对这个时间线有顾虑。

场景上下文让记忆更丰富、召回更精准。当你以后问"上次那个会是什么情况"，场景记忆能给出比文字记忆丰富得多的答案。

**3. AI 能主动发起交互。**

OpenClaw 只能在你发消息时被动回复（或定时提醒）。它不知道你当前在做什么，所以无法判断"现在是不是一个合适的时机提醒你"。

你的设备持续感知你的状态。它知道你在专注写代码（不要打扰），还是在喝水休息（可以提醒），还是刚结束一个会议（可以总结）。它能在合适的时机主动开口，而不是盲目推送通知。

## 推演三：如果这个框架能扩展到其他物理形态

现在这个框架跑在一台桌面设备上。但它的核心组件——感知管线、快慢认知、记忆系统、工具调用、事件总线——这些是**跟具体硬件无关的**。

推演几种不同的物理形态：

### 桌面设备（现在）

放在你办公桌上。摄像头看到你和你面前的东西，麦克风听到你说的话和周围的对话。

核心价值：工作场景的环境感知 + 记忆。

### 会议室设备

放在会议室桌子中间。广角摄像头看到所有参会者，麦克风阵列采集所有人的声音。

核心价值：自动会议记录。不是录音转写——是理解谁说了什么、做了什么决定、有什么待办。会后任何人可以问它"我们刚才决定了什么"。

这个场景下设备售价 $3000-5000 对企业来说完全合理。一个 10 人团队每天开 2 小时会，如果这台设备省掉每个人 15 分钟的会后整理时间，一个月就回本。

### 车载设备

放在车里。摄像头看到车内（驾驶员状态、乘客），麦克风听到车内对话。

核心价值：车内对话记忆（"刚才你老婆说的那个餐厅叫什么？"）+ 驾驶员状态感知（疲劳检测）+ 语音助手。

### 家庭设备

放在客厅或厨房。看到家庭场景，听到家庭对话。

核心价值：家庭生活记忆（"上次医生说孩子的药该怎么吃来着？"）+ 智能家居控制 + 家庭助手。

### 可穿戴

更小的硬件（未来），像眼镜或胸针。第一人称视角。

核心价值：你在外面见客户、逛展会、参加活动，设备从你的视角采集信息。所有场景都跟着你，不局限于桌面。

### 机器人

把同一套框架跑在一个有轮子/腿的移动平台上。

核心价值：具身智能。它不只是感知——它可以物理地与环境互动。仓库管理、巡检、陪伴、服务。

**关键洞察：所有这些形态用的是同一套软件框架。**

```
具体硬件不同        共同的框架层                具体场景不同
─────────────    ─────────────────────    ────────────────
桌面摄像头    ┐                          ┌  办公记忆
会议室阵列    │   PerceptionProvider      │  会议纪要
车载摄像头    ├→  ReasoningProvider     →──┤  车内助手
可穿戴相机    │   MemoryStore             │  个人记录
机器人传感器  ┘   ToolRegistry            └  具身操作
                 EventBus
                 GPUScheduler
```

PerceptionProvider 的输入从桌面摄像头变成机器人的 LIDAR+摄像头，换一个 Provider 实现就行。ReasoningProvider 不变。MemoryStore 不变。EventBus 不变。

**这就是"框架"的意义：你现在写的代码不只是给一台桌面设备用的，而是给所有"有感知、有认知、有记忆"的物理 AI 系统用的。**

## 推演四：OpenClaw 的位置和你的位置

```
                    纯虚拟                    有物理感知
                ┌──────────────────────┬──────────────────────┐
                │                      │                      │
  云端大脑      │  OpenClaw            │  某些云端机器人       │
                │  LangChain           │  (感知在端侧,        │
                │  AutoGPT             │   推理在云端)         │
                │  (成熟,生态丰富,      │                      │
                │   但无感知,不私有)     │                      │
                │                      │                      │
                ├──────────────────────┼──────────────────────┤
                │                      │                      │
  本地大脑      │  Ollama / LMStudio   │  ← 你在这里           │
                │  (只有推理,           │  (框架+感知+本地      │
                │   没有框架和感知)      │   目前唯一)           │
                │                      │                      │
                └──────────────────────┴──────────────────────┘
```

- 左上：OpenClaw 们。框架成熟，生态丰富，但没有物理感知，大脑在云端。
- 右上：一些云端机器人方案。有感知，但推理依赖云端。
- 左下：Ollama 们。本地推理，但没有框架、没有感知。
- **右下：你。本地大脑 + 框架 + 物理感知。目前这个象限基本是空的。**

为什么右下角是空的？因为它需要三个条件同时满足：
1. 硬件足够强（Orin 64GB 级别），一年前不存在消费级方案
2. 本地多模态模型足够好（MiniCPM-o），一年前没有
3. 本地推理模型足够强（Qwen3.5 27B），两年前不行

三个条件 2025-2026 年刚好同时成熟。你是第一批站到这个位置的人。

## 推演五：框架层面你的代码已经有什么、还缺什么

对照 OpenClaw 的框架能力，看你现在的代码：

| 能力 | OpenClaw | 你的代码 | 状态 |
|---|---|---|---|
| Provider 抽象 | 支持 Claude/GPT/Ollama 切换 | PerceptionProvider + ReasoningProvider | ✅ 已有 |
| 记忆系统 | auto-capture, auto-recall, semantic search | MEMORY.md + 关键词 grep | ⚠️ 基础版，需要加语义检索和自动采集 |
| 工具调用 | 丰富的内置工具 | search/file/shell/computer/memory_save | ✅ 已有 |
| 通道系统 | WhatsApp/Telegram/Discord/CLI | CLI + WebSocket | ⚠️ 有框架，通道种类少 |
| 人格系统 | 角色定义 | SOUL.md | ✅ 已有 |
| 事件驱动 | 不强调 | EventBus | ✅ 已有（且比 OpenClaw 更适合感知场景） |
| **多模态感知** | **没有** | **MiniCPM-o** | **✅ 你有，OpenClaw 没有** |
| **快慢认知** | **没有** | **GPUScheduler + 双系统** | **✅ 你有，OpenClaw 没有** |
| **物理设备支持** | **没有** | **硬件集成** | **✅ 你有，OpenClaw 没有** |

你和 OpenClaw 的框架能力其实已经很接近了。你缺的主要是：
1. 记忆系统的成熟度（语义检索、auto-capture、auto-recall）
2. 更多的通道支持

你有而 OpenClaw 永远不会有的：
1. 多模态感知管线（因为 OpenClaw 是虚拟的，没有传感器）
2. 快慢认知架构（因为 OpenClaw 不需要持续感知）
3. 物理设备层（因为 OpenClaw 不跑在专用硬件上）

## 推演七：快慢系统在框架层面的意义

之前讨论快慢系统都是从单台设备的角度。从框架角度看，意义更大。

**任何有物理感知的 AI 系统都需要快慢分离。**

- 机器人：感知层持续处理摄像头/LIDAR/触觉传感器（快），规划层做路径规划和决策（慢）
- 自动驾驶：感知层持续检测车道/行人/交通信号（快），决策层做变道/制动/加速判断（慢）
- 你的桌面设备：感知层持续理解音视频（快），推理层做记忆管理和用户响应（慢）

快慢分离是所有具身 AI 系统的共同架构。你在桌面设备上实现的 GPUScheduler + PerceptionProvider + ReasoningProvider，本质上是在实现这个通用范式。

这意味着你做的不只是"让两个模型在一块 GPU 上交替运行"。你做的是**具身 AI 的标准认知架构**——感知-推理分离、异步事件驱动、资源调度。

任何未来接入你框架的具身设备都可以复用这套架构。

## MiniCPM-o 在框架里的真正角色

MiniCPM-o 不是"一个可选的感知模型"。在你的框架里它代表的是一种**能力范式**：

**端侧实时多模态理解。**

今天是 MiniCPM-o。明年可能是 MiniCPM-o 的后继者，或者另一个更好的端侧多模态模型。模型会换代，但"端侧实时多模态理解"这个能力槽位不会消失。

在你的框架里，这个槽位就是 PerceptionProvider 接口。MiniCPM-o 是目前填这个槽位最好的实现。未来有更好的模型，换一个 Provider 实现就行——框架不需要改。

这也是为什么你的架构设计（Provider 抽象 + EventBus + GPUScheduler）是正确的：

- PerceptionProvider 抽象了"快感知"的能力，不绑定具体模型
- ReasoningProvider 抽象了"慢推理"的能力，不绑定具体模型
- EventBus 解耦了感知和推理，两者通过事件通信
- GPUScheduler 管理两者对硬件资源的竞争

**这套抽象在模型换代时不需要重写。它是稳定的框架层。**

## 产品价值总结

回到最初的问题：价值和意义到底在哪？

### 短期价值（这台设备）

> 一台放在你桌上的 AI，它持续看、持续听、理解发生了什么、记住重要的事。你随时可以问它，它也会在合适的时候主动告诉你。所有数据不出设备。

这件事没有任何其他产品做到。不是因为技术做不到，而是云端 AI 过不了隐私关，纯软件方案做不了 always-on 感知，其他框架没有多模态感知能力。

### 长期价值（这个框架）

> 第一个开源的具身 AI Agent 框架。不是给聊天机器人用的，是给有眼睛、有耳朵、有身体的 AI 系统用的。

OpenClaw 做了虚拟 AI Agent 的框架，339k stars，证明了框架层面的需求。你在做物理 AI Agent 的框架——同样的框架能力（记忆、工具、通道、人格），加上 OpenClaw 不可能有的感知管线和认知架构。

桌面设备是第一个实例。会议室设备、家庭设备、可穿戴、机器人是未来的实例。框架是不变的底层。

### 快慢系统的价值

不是一个设计选择。是任何具身 AI 系统的物理约束。你现在在一台桌面设备上解决了这个问题（GPUScheduler），这套方案对所有未来的具身 AI 形态都有效。

### MiniCPM-o 的价值

不是"一个模型"。是框架里"端侧实时感知"这个能力槽位的当前最佳实现。它让设备从"语音终端"变成"有感知的实体"。模型会换代，但这个槽位是永久的。

## 怎么打

### 定位

不要定位为"一台 AI 设备"。定位为：

> **面向具身 AI 的开源 Agent 框架。桌面设备是第一个参考实现。**

这个定位同时覆盖了：
- 短期：你自己的桌面设备（证明框架有效）
- 中期：其他开发者用你的框架造他们的设备
- 长期：不同物理形态的 AI 系统

### 路径

**第一阶段（现在 → 8 周）：把桌面设备做到可用**

1. 打通感知链路：MiniCPM-o 持续处理音视频 → 输出感知事件 → EventBus
2. 打通记忆链路：感知事件 → Qwen3.5 判断 → 自动写入 MEMORY.md
3. 实现语义检索：embedding 模型 + 向量匹配
4. 实现主动提醒：WebSocket send + 语音输出
5. 自己每天用，积累真实记忆和反馈

**第二阶段（8 周 → 16 周）：框架化 + 开源**

6. 把代码整理成清晰的框架层：Provider 接口、EventBus、MemoryStore、ToolRegistry
7. 写开发者文档：怎么实现自己的 PerceptionProvider、怎么添加新工具、怎么定义新通道
8. 发布硬件 BOM + 组装指南
9. 录 demo 视频：
   - 视频 A（产品向）：展示"设备在场"的体验——正常工作、设备自动记忆、事后精确召回
   - 视频 B（框架向）：展示框架的扩展性——换一个 Provider、加一个工具、换一个硬件形态
10. 发 Hacker News / Twitter / B 站

**第三阶段（16 周+）：社区 + 生态**

11. 建立开发者社区
12. 吸引贡献者做新的 Provider 实现、通道插件、工具插件
13. 有人在不同硬件/场景上跑你的框架
14. 从社区反馈中找到最有价值的场景，深耕

### demo 应该展示什么

两个维度的冲击力：

**维度一：它在场（产品冲击力）**

你正常工作。设备在旁边。你跟同事讨论问题。事后你问设备"刚才说了什么"。它精确回答，而且知道当时的场景上下文。

"我什么都没操作，但它知道发生了什么。"

**维度二：它是框架（技术冲击力）**

展示同一套代码跑在不同场景下：桌面设备听你工作、会议室设备记录会议、同一个记忆系统被多个通道访问。

"这不是一个 App，这是一个平台。"

## 风险

**风险一：框架叙事太早。** 在桌面设备的体验还没做好之前就谈"具身 AI 框架"，容易变成空话。
缓解：先让一台设备做到"惊艳"。框架叙事在有了可工作的参考实现之后再讲。代码可以提前按框架结构写，但对外宣传先聚焦设备体验。

**风险二：MiniCPM-o 的持续感知模式未验证。** 它是为对话设计的，不是为被动观察设计的。长时间持续运行时的质量、功耗、稳定性都未知。
缓解：这是第一步要验证的事。如果 MiniCPM-o 做不了持续被动感知，需要混合方案（Whisper 持续 ASR + MiniCPM-o 按需多模态理解）。

**风险三：记忆质量。** 自动采集的记忆是否有用，取决于 AI 判断"什么值得记"的准确度。27B 模型是否够用，只有实测知道。
缓解：初期以显式记忆为主，被动采集逐步放开，让用户能调阈值。

**风险四：$3000 硬件门槛。** 限制了早期用户数量。
缓解：框架是软件，不绑定 Orin。有 GPU 的开发者可以在自己电脑上跑。等硬件成本下降（GB10 等），更多人能用。
