# gemma4:26b vs qwen3.5:35b 深度对比报告

> 测试时间: 2026-04-03
> 硬件: DGX Spark (GB10, 128GB LPDDR5x, 273 GB/s, CUDA 13.0)
> 引擎: Ollama 0.20.0
> 量化: 均为 Q4_K_M

## 模型概况

| 项目 | gemma4:26b | qwen3.5:35b |
|------|-----------|-------------|
| 厂商 | Google | Alibaba |
| 架构 | MoE | MoE |
| 总参数 | 26B | 35B |
| 激活参数 | **4B** | **3B** |
| 模型大小 | **18GB** | 23GB |
| 上下文窗口 | **256K** | 131K |
| 多模态 | **视觉 + 音频** | 纯文本 |
| Thinking 模式 | 无 | 默认开启 |
| 发布时间 | 2026-04 | 2025-06 |

## 一、速度对比

### tok/s（生成速率）

| Prompt | gemma4:26b | qwen3.5:35b | 差距 |
|--------|-----------|-------------|------|
| hello | 37.9 | **54.3** | -30% |
| reasoning | 37.5 | **56.0** | -33% |
| code | 33.7 | **56.5** | -40% |
| long_output | 34.3 | **55.9** | -39% |
| chinese | 32.0 | **56.1** | -43% |
| tool_weather | 36.4 | **56.7** | -36% |
| tool_multi | 36.0 | **57.0** | -37% |
| **平均** | **35.1** | **55.8** | **-37%** |

**qwen3.5:35b 的 tok/s 快 59%。** 根本原因：qwen 只激活 3B 参数，gemma4 激活 4B，在带宽瓶颈（273 GB/s）下，少 25% 的激活参数直接转化为更高的吞吐。

### TTFT（首 token 延迟）

| Prompt | gemma4:26b | qwen3.5:35b | 差距 |
|--------|-----------|-------------|------|
| hello | **0.03s** | 0.10s | **3.3x 快** |
| reasoning | 0.10s | 0.09s | ≈持平 |
| code | **0.08s** | 0.09s | 略快 |
| long_output | **0.06s** | 0.09s | **1.5x 快** |
| chinese | **0.07s** | 0.09s | 略快 |
| tool_weather | **0.11s** | 0.27s | **2.5x 快** |
| tool_multi | **0.11s** | 0.32s | **2.9x 快** |

**gemma4:26b 的 TTFT 全面领先**，尤其工具调用场景快 2-3 倍。qwen3.5 的 thinking 初始化会额外消耗首 token 时间。

### 端到端响应时间（wall time）— 用户真正的等待时间

| Prompt | gemma4:26b | qwen3.5:35b | 胜者 |
|--------|-----------|-------------|------|
| hello | **3.9s** | 20.2s | **gemma4 快 5.2x** |
| reasoning | **11.5s** | 39.2s | **gemma4 快 3.4x** |
| code | 53.4s | **10.8s** | qwen 快 4.9x |
| long_output | 38.9s | **37.8s** | ≈持平 |
| chinese | 66.3s | **53.8s** | qwen 快 1.2x |
| tool_weather | 2.1s | 2.1s | 持平 |
| tool_multi | 2.4s | 2.0s | ≈持平 |

**惊人反转：** 虽然 qwen 的 tok/s 快 59%，但端到端响应时间 gemma4 在 hello 和 reasoning 上分别快 **5.2 倍和 3.4 倍**。

原因：qwen3.5 的 thinking 模式会生成大量不可见的思考 token（hello 产生 449 token，gemma4 只产生 138 token），真正有效信息的比例不同。

### 输出 token 数对比

| Prompt | gemma4:26b | qwen3.5:35b | 倍数 |
|--------|-----------|-------------|------|
| hello | 138 | 449 | qwen 输出 3.3x |
| reasoning | 414 | 2,147 | qwen 输出 5.2x |
| code | 1,774 | 587 | **gemma 输出 3.0x** |
| long_output | 1,312 | 2,068 | qwen 输出 1.6x |
| chinese | 2,093 | 2,957 | qwen 输出 1.4x |

qwen3.5 在多数场景下生成的 token 多 1.5-5 倍（因 thinking 链），但代码生成反而是 gemma4 更详细。

两者算法一致，gemma4 提供了更多解释和测试用例，qwen3.5 代码更精炼。

## 三、效率对比

| 指标 | gemma4:26b | qwen3.5:35b | 胜者 |
|------|-----------|-------------|------|
| tok/s | 35.1 | 55.8 | qwen |
| 模型大小 | 18GB | 23GB | **gemma4** (省 22%) |
| tok/s per GB | 1.95 | 2.43 | qwen |
| 有效输出/总token | ~90% | ~30-50% | **gemma4** |
| 端到端效率 (hello) | 3.9s | 20.2s | **gemma4** |

gemma4 的"真实效率"更高：每个 token 都是有效输出，没有被 thinking 消耗。

## 四、各场景推荐

| 场景 | 推荐 | 原因 |
|------|------|------|
| 快速问答/闲聊 | **gemma4:26b** | 端到端 3.9s vs 20.2s，体感差距巨大 |
| 深度代码生成 | **qwen3.5:35b** | thinking 模式帮助推理，代码精炼 |
| Agent/工具调用 | **gemma4:26b** | TTFT 快 2-3 倍，省内存，原生支持 |
| 长文写作 | 平手 | 耗时接近，质量相当 |
| 中文对话 | 平手 | 两者中文能力都优秀 |
| 多模态（图片/音频） | **gemma4:26b** | qwen3.5:35b 不支持 |
| 批量/后台处理 | **qwen3.5:35b** | 纯 tok/s 更高，适合吞吐优先 |
| 内存敏感（并发） | **gemma4:26b** | 18GB vs 23GB，省出空间跑其他模型 |

## 五、总结

**表面上**，qwen3.5:35b 以 55.8 tok/s 大幅领先 gemma4:26b 的 35.1 tok/s（快 59%）。

**实际上**，在用户体感最重要的"端到端响应时间"维度，gemma4:26b 在对话场景下快了 **3-5 倍**，因为：
1. 不生成 thinking token，输出即所见
2. TTFT 更低（无 thinking 初始化开销）
3. 回答更精炼，不浪费 token

**如果只能选一个模型：**
- 追求**交互体验**（对话、Agent）→ **gemma4:26b**
- 追求**推理深度**（复杂代码、数学）→ **qwen3.5:35b**
- 两者共存完全可行：18GB + 23GB = 41GB，128GB 内存绰绰有余
